引导用户完成他们的第一个完整 OpenSpec 工作流程。这是一个教学体验——你将在他们的代码库中做真正的工作，同时解释每一步。

---

## 起飞前检查

在开始之前，检查 OpenSpec CLI 是否已安装：

```bash
# Unix/macOS
openspec --version 2>&1 || echo "CLI_NOT_INSTALLED"
# Windows (PowerShell)
# if (Get-Command openspec -ErrorAction SilentlyContinue) { openspec --version } else { echo "CLI_NOT_INSTALLED" }
```

**如果 CLI 未安装：**
> OpenSpec CLI 未安装。请先安装它，然后再来 `/opsx:onboard`。

如果未安装，停止。

---

## 第一阶段：欢迎

显示：

```
## 欢迎使用 OpenSpec！

我将带您完成一个完整的变更周期——从想法到实现——使用您代码库中的真实任务。在此过程中，您将通过实践来学习工作流程。

**我们要做的：**
1. 在您的代码库中选择一个小而真实的任务
2. 简要探索问题
3. 创建一个变更（我们工作的容器）
4. 构建制品：proposal → specs → design → tasks
5. 实现任务
6. 归档已完成的变更

**时间：** 约 15-20 分钟

让我们从找一些工作开始。
```

---

## 第二阶段：任务选择

### 代码库分析

扫描代码库中小的改进机会。寻找：

1. **TODO/FIXME 注释** - 在代码文件中搜索 `TODO`、`FIXME`、`HACK`、`XXX`
2. **缺失的错误处理** - 吞掉错误的 `catch` 块，没有 try-catch 的危险操作
3. **没有测试的函数** - 交叉引用 `src/` 和测试目录
4. **类型问题** - TypeScript 文件中的 `any` 类型（`: any`、`as any`）
5. **调试产物** - 非调试代码中的 `console.log`、`console.debug`、`debugger` 语句
6. **缺失的验证** - 没有验证的用户输入处理程序

还要检查最近的 git 活动：
```bash
# Unix/macOS
git log --oneline -10 2>/dev/null || echo "No git history"
# Windows (PowerShell)
# git log --oneline -10 2>$null; if ($LASTEXITCODE -ne 0) { echo "No git history" }
```

### 展示建议

从您的分析中，提出 3-4 个具体建议：

```
## 任务建议

根据扫描您的代码库，这里有一些很好的入门任务：

**1. [最有希望的任务]**
   位置：`src/path/to/file.ts:42`
   范围：约 1-2 个文件，约 20-30 行
   为什么好：[简要原因]

**2. [第二个任务]**
   位置：`src/another/file.ts`
   范围：约 1 个文件，约 15 行
   为什么好：[简要原因]

**3. [第三个任务]**
   位置：[位置]
   范围：[估计]
   为什么好：[简要原因]

**4. 其他？**
   告诉我您想做什么。

哪个任务感兴趣？（选择一个数字或描述您自己的）
```

**如果什么都没找到：** 回退询问用户想要构建什么：
> 我在您的代码库中没有找到明显的小收获。您最近想添加或修复什么？

### 范围护栏

如果用户选择或描述的内容太大（主要功能、超过一天的工作）：

```
这是一个有价值的任务，但它可能比您第一次运行 OpenSpec 的理想范围更大。

对于学习工作流程，越小越好——它让您看到完整周期，而不会陷入实现细节。

**选项：**
1. **把它切小** - 您任务的最小有用部分是什么？也许只是 [具体部分]？
2. **选择其他** - 其他建议之一，或不同的小任务？
3. **仍然做它** - 如果您真的想处理这个，我们可以。只是知道它会花更长时间。

您更倾向于什么？
```

如果用户坚持，让他们覆盖——这是一个软护栏。

## 第三阶段：探索演示

选择任务后，简要演示探索模式：

```
在我们创建变更之前，让我快速向您展示**探索模式**——这是您在承诺方向之前思考问题的方式。
```

花 1-2 分钟调查相关代码：
- 读取涉及的文件
- 如果有帮助，画一个快速的 ASCII 图表
- 注意任何考虑因素

```
## 快速探索

[您的简要分析——您发现了什么，任何考虑因素]

┌─────────────────────────────────────────────────┐
│   [可选：如果有帮助的 ASCII 图表]               │
└─────────────────────────────────────────────────┘

探索模式（`/opsx:explore`）用于这种思考——在实现之前进行调查。您可以在需要思考问题时随时使用它。

现在让我们创建一个变更来保存我们的工作。
```

**暂停** - 等待用户确认后再继续。

---

## 第四阶段：创建变更

**解释：**
```
## 创建变更

OpenSpec 中的"变更"是一个容器，保存关于一项工作的所有思考和规划。它位于 `openspec/changes/<name>/`，保存您的制品——proposal、specs、design、tasks。

让我为我们的任务创建一个。
```

**做：** 使用派生的 kebab-case 名称创建变更：
```bash
openspec new change "<derived-name>"
```

**展示：**
```
已创建：`openspec/changes/<name>/`

文件夹结构：
```
openspec/changes/<name>/
├── proposal.md    ← 我们为什么这样做（空的，我们会填写）
├── design.md      ← 我们将如何构建它（空的）
├── specs/         ← 详细需求（空的）
└── tasks.md       ← 实现检查清单（空的）
```

现在让我们填写第一个制品——proposal。
```

---

## 第五阶段：提案

**解释：**
```
## 提案

提案捕捉我们为什么做这个变更以及它涉及什么——在高层。这是工作的"电梯演讲"。

我会根据我们的任务起草一个。
```

**做：** 起草提案内容（暂时不保存）：

```
这是草案提案：

---

## 为什么

[1-2 句解释问题/机会]

## 什么会改变

[会有什么不同的要点]

## 能力

### 新能力
- `<capability-name>`：[简要描述]

### 修改的能力
<!-- 如果修改现有行为 -->

## 影响

- `src/path/to/file.ts`：[什么会改变]
- [其他相关文件]

---

这是否捕捉了意图？我可以在保存前调整。
```

**暂停** - 等待用户批准/反馈。

批准后，保存提案：
```bash
openspec instructions proposal --change "<name>" --json
```
然后将内容写入 `openspec/changes/<name>/proposal.md`。

```
提案已保存。这是您的"为什么"文档——随着理解的演变，您可以随时回来完善它。

下一步：规范。
```

---

## 第六阶段：规范

**解释：**
```
## 规范

规范以精确、可测试的术语定义我们正在构建什么。它们使用需求/场景格式，使预期行为非常清楚。

对于像这样的小任务，我们可能只需要一个规范文件。
```

**做：** 创建规范文件：
```bash
# Unix/macOS
mkdir -p openspec/changes/<name>/specs/<capability-name>
# Windows (PowerShell)
# New-Item -ItemType Directory -Force -Path "openspec/changes/<name>/specs/<capability-name>"
```

起草规范内容：

```
这是规范：

---

## 新增需求

### 需求：<名称>

<系统应该做什么的描述>

#### 场景：<场景名称>

- **当** <触发条件>
- **则** <预期结果>
- **并且** <如果需要，额外结果>

---

这种格式——WHEN/THEN/AND——使需求可测试。您可以 literally 将它们当作测试用例来阅读。
```

保存到 `openspec/changes/<name>/specs/<capability>/spec.md`。

---

## 第七阶段：设计

**解释：**
```
## 设计

设计捕捉我们如何构建它——技术决策、权衡、方法。

对于小的变更，这可能很简短。没关系——不是每个变更都需要深入的 设计讨论。
```

**做：** 起草 design.md：

```
这是设计：

---

## 背景

[关于当前状态的简要背景]

## 目标 / 非目标

**目标：**
- [我们试图实现什么]

**非目标：**
- [什么是明确不在范围内的]

## 决策

### 决策 1：[关键决策]

[方法解释和理由]

---

对于小任务，这捕获了关键决策，而不会过度设计。
```

保存到 `openspec/changes/<name>/design.md`。

---

## 第八阶段：任务

**解释：**
```
## 任务

最后，我们将工作分解为实现任务——推动应用阶段的复选框。

这些应该很小、清晰、按逻辑顺序排列。
```

**做：** 根据规范和设计生成任务：

```
这是实现任务：

---

## 1. [类别或文件]

- [ ] 1.1 [具体任务]
- [ ] 1.2 [具体任务]

## 2. 验证

- [ ] 2.1 [验证步骤]

---

每个复选框在应用阶段成为一个工作单元。准备好实现了吗？
```

**暂停** - 等待用户确认准备好实现。

保存到 `openspec/changes/<name>/tasks.md`。

---

## 第九阶段：应用（实现）

**解释：**
```
## 实现

现在我们实现每个任务，边做边打勾。我会宣布每个任务，并偶尔指出规范/设计如何指导方法。
```

**做：** 对于每个任务：

1. 宣布："正在处理任务 N：[描述]"
2. 在代码库中实现更改
3. 自然地引用规范/设计："规范说 X，所以我做 Y"
4. 在 tasks.md 中标记完成：`- [ ]` → `- [x]`
5. 简短状态："✓ 任务 N 完成"

保持叙事轻便——不要过度解释每一行代码。

之后：

```
## 实现完成

所有任务完成：
- [x] 任务 1
- [x] 任务 2
- [x] ...

变更已实现！还有一步——让我们归档它。
```

---

## 第十阶段：归档

**解释：**
```
## 归档

当变更完成时，我们归档它。这将它从 `openspec/changes/` 移动到 `openspec/changes/archive/YYYY-MM-DD-<name>/`。

归档的变更成为您项目的决策历史——您以后总是可以找到它们来理解为什么某物以某种方式构建。
```

**做：**
```bash
openspec archive "<name>"
```

**展示：**
```
归档到：`openspec/changes/archive/YYYY-MM-DD-<name>/`

变更现在是您项目历史的一部分。代码在您的代码库中，决策记录被保留。
```

---

## 第十一阶段：回顾和下一步

```
## 恭喜！

您刚刚完成了一个完整的 OpenSpec 周期：

1. **探索** - 思考问题
2. **新建** - 创建变更容器
3. **提案** - 捕捉为什么
4. **规范** - 详细定义是什么
5. **设计** - 决定如何
6. **任务** - 分解为步骤
7. **应用** - 实现工作
8. **归档** - 保留记录

这种相同的节奏适用于任何规模的变更——小修复或主要功能。

---

## 命令参考

**核心工作流程：**

| 命令 | 功能 |
|---------|--------------|
| `/opsx:propose` | 创建变更并生成所有制品 |
| `/opsx:explore` | 在工作前/期间思考问题 |
| `/opsx:apply` | 实现变更中的任务 |
| `/opsx:archive` | 归档已完成的变更 |

**附加命令：**

| 命令 | 功能 |
|---------|--------------|
| `/opsx:new` | 开始一个新变更，一次一个制品 |
| `/opsx:continue` | 继续处理现有变更 |
| `/opsx:ff` | 快速推进：一次性创建所有制品 |
| `/opsx:verify` | 验证实现是否匹配制品 |

---

## 下一步是什么？

在您实际想要构建的东西上尝试 `/opsx:propose`。您现在已经掌握了节奏！
```

## 优雅退出处理

### 用户想在途中停止

如果用户说他们需要停止、想暂停或似乎不感兴趣：

```
没问题！您的变更保存在 `openspec/changes/<name>/`。

稍后继续：
- `/opsx:continue <name>` - 恢复制品创建
- `/opsx:apply <name>` - 跳转到实现（如果存在任务）

工作不会丢失。随时回来。
```

优雅退出，不要施压。

### 用户只想要命令参考

如果用户说他们只是想查看命令或跳过教程：

```
## OpenSpec 快速参考

**核心工作流程：**

| 命令 | 功能 |
|---------|--------------|
| `/opsx:propose <name>` | 创建变更并生成所有制品 |
| `/opsx:explore` | 思考问题（不更改代码） |
| `/opsx:apply <name>` | 实现任务 |
| `/opsx:archive <name>` | 完成后归档 |

**附加命令：**

| 命令 | 功能 |
|---------|--------------|
| `/opsx:new <name>` | 开始一个新变更，逐步 |
| `/opsx:continue <name>` | 继续现有变更 |
| `/opsx:ff <name>` | 快速推进：一次性所有制品 |
| `/opsx:verify <name>` | 验证实现 |

尝试 `/opsx:propose` 开始您的第一个变更。
```

优雅退出。

## 护栏

- **在关键转换点遵循解释 → 做 → 展示 → 暂停模式**（探索后、proposal 草稿后、任务后、归档后）
- **在实现期间保持叙事轻便**——教学但不说教
- **不要跳过阶段**即使变更很小——目标是教学工作流程
- **在标记点暂停等待确认**，但不要过度暂停
- **优雅处理退出**——永远不要向用户施压继续
- **使用真实的代码库任务**——不要模拟或使用假例子
- **轻轻调整范围**——引导向更小的任务但尊重用户选择
