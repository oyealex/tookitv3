一次性归档多个已完成的变更。

此技能允许您批量归档变更，通过检查代码库来智能处理规范冲突，以确定实际实现了什么。

**输入**：无需输入（提示选择）

**步骤**

1. **获取活动变更**

   运行 `openspec list --json` 获取所有活动变更。

   如果不存在活动变更，通知用户并停止。

2. **提示选择变更**

   使用 **AskUserQuestion 工具** 的多选功能让用户选择变更：
   - 显示每个变更及其模式
   - 包含"所有变更"选项
   - 允许任意数量的选择（1+ 可以，2+ 是典型用例）

   **重要**：不要自动选择。始终让用户选择。

3. **批量验证 - 收集所有选定变更的状态**

   对于每个选定的变更，收集：

   a. **制品状态** - 运行 `openspec status --change "<name>" --json`
      - 解析 `schemaName` 和 `artifacts` 列表
      - 注意哪些制品是 `done` vs 其他状态

   b. **任务完成** - 读取 `openspec/changes/<name>/tasks.md`
      - 计算 `- [ ]`（未完成）vs `- [x]`（完成）
      - 如果不存在任务文件，记录为"No tasks"

   c. **增量规范** - 检查 `openspec/changes/<name>/specs/` 目录
      - 列出存在哪些能力规范
      - 对于每个，提取需求名称（匹配 `### Requirement: <name>` 的行）

4. **检测规范冲突**

   构建 `capability -> [触及它的变更]` 的映射：

   ```
   auth -> [change-a, change-b]  <- 冲突（2+ 个变更）
   api  -> [change-c]            <- 正常（只有 1 个变更）
   ```

   当 2+ 个选定的变更具有相同能力的增量规范时，存在冲突。

5. **智能解决冲突**

   **对于每个冲突**，调查代码库：

   a. **读取冲突变更的增量规范**以了解每个声称添加/修改的内容

   b. **搜索代码库**以获取实现证据：
      - 查找实现每个增量规范需求的相关代码
      - 检查相关文件、函数或测试

   c. **确定解决方案**：
      - 如果只有一个变更实际被实现 -> 同步该变更的规范
      - 如果两者都被实现 -> 按时间顺序应用（较旧的在前，较新的覆盖）
      - 如果都未实现 -> 跳过规范同步，提醒用户

   d. **记录每个冲突的解决方案**：
      - 应用哪个变更的规范
      - 以什么顺序（如果两者都应用）
      - 理由（在代码库中发现了什么）

6. **显示合并状态表**

   显示总结所有变更的表：

   ```
   | 变更               | 制品   | 任务 | 规范   | 冲突   | 状态   |
   |---------------------|-----------|-------|---------|-----------|--------|
   | schema-management   | 完成      | 5/5   | 2 增量 | 无        | 就绪   |
   | project-config      | 完成      | 3/3   | 1 增量 | 无        | 就绪   |
   | add-oauth           | 完成      | 4/4   | 1 增量 | auth (!)  | 就绪*  |
   | add-verify-skill    | 1 剩余    | 2/5   | 无     | 无        | 警告   |
   ```

   对于冲突，显示解决方案：
   ```
   * 冲突解决方案：
     - auth 规范：将先应用 add-oauth 然后 add-jwt（两者都已实现，按时间顺序）
   ```

   对于未完成的变更，显示警告：
   ```
   警告：
   - add-verify-skill: 1 个未完成制品，3 个未完成任务
   ```

7. **确认批量操作**

   使用 **AskUserQuestion 工具** 进行单一确认：

   - "归档 N 个变更？"并提供基于状态的选项
   - 选项可能包括：
     - "归档所有 N 个变更"
     - "只归档 N 个就绪变更（跳过未完成的）"
     - "取消"

   如果存在未完成的变更，明确说明它们将带警告归档。

8. **为每个确认的变更执行归档**

   按确定的顺序处理变更（尊重冲突解决方案）：

   a. **同步规范**如果存在增量规范：
      - 使用 openspec-sync-specs 方法（代理驱动的智能合并）
      - 对于冲突，按已解决顺序应用
      - 跟踪是否完成同步

   b. **执行归档**：
      ```bash
      mkdir -p openspec/changes/archive
      mv openspec/changes/<name> openspec/changes/archive/YYYY-MM-DD-<name>
      ```

   c. **跟踪每个变更的结果**：
      - 成功：归档成功
      - 失败：归档期间出错（记录错误）
      - 跳过：用户选择不归档（如果适用）

9. **显示摘要**

   显示最终结果：

   ```
   ## 批量归档完成

   已归档 3 个变更：
   - schema-management-cli -> archive/2026-01-19-schema-management-cli/
   - project-config -> archive/2026-01-19-project-config/
   - add-oauth -> archive/2026-01-19-add-oauth/

   跳过 1 个变更：
   - add-verify-skill（用户选择不归档未完成的）

   规范同步摘要：
   - 4 个增量规范已同步到主规范
   - 1 个冲突已解决（auth：按时间顺序应用了两者）
   ```

   如果有任何失败：
   ```
   失败 1 个变更：
   - some-change: 归档目录已存在
   ```

**冲突解决方案示例**

示例 1：只有一个实现
```
冲突：specs/auth/spec.md 被 [add-oauth, add-jwt] 触及

检查 add-oauth：
- 增量添加了 "OAuth Provider Integration" 需求
- 搜索代码库... 找到了实现 OAuth 流程的 src/auth/oauth.ts

检查 add-jwt：
- 增量添加了 "JWT Token Handling" 需求
- 搜索代码库... 找不到 JWT 实现

解决方案：只有 add-oauth 已实现。将只同步 add-oauth 规范。
```

示例 2：两者都实现
```
冲突：specs/api/spec.md 被 [add-rest-api, add-graphql] 触及

检查 add-rest-api（创建于 2026-01-10）：
- 增量添加了 "REST Endpoints" 需求
- 搜索代码库... 找到了 src/api/rest.ts

检查 add-graphql（创建于 2026-01-15）：
- 增量添加了 "GraphQL Schema" 需求
- 搜索代码库... 找到了 src/api/graphql.ts

解决方案：两者都已实现。将先应用 add-rest-api 规范，
然后应用 add-graphql 规范（按时间顺序，较新的优先）。
```

**成功时的输出**

```
## 批量归档完成

已归档 N 个变更：
- <change-1> -> archive/YYYY-MM-DD-<change-1>/
- <change-2> -> archive/YYYY-MM-DD-<change-2>/

规范同步摘要：
- N 个增量规范已同步到主规范
- 无冲突（或：已解决 M 个冲突）
```

**部分成功时的输出**

```
## 批量归档完成（部分）

已归档 N 个变更：
- <change-1> -> archive/YYYY-MM-DD-<change-1>/

跳过 M 个变更：
- <change-2>（用户选择不归档未完成的）

失败 K 个变更：
- <change-3>: 归档目录已存在
```

**没有变更时的输出**

```
## 没有要归档的变更

未找到活动变更。创建新变更以开始。
```

**护栏**
- 允许任意数量的变更（1+ 可以，2+ 是典型用例）
- 始终提示选择，永不自动选择
- 尽早检测规范冲突并通过检查代码库解决
- 当两个变更都已实现时，按时间顺序应用规范
- 只在实现缺失时跳过规范同步（提醒用户）
- 在确认前显示每个变更的清晰状态
- 对整个批次使用单一确认
- 跟踪并报告所有结果（成功/跳过/失败）
- 移动到归档时保留 .openspec.yaml
- 归档目录目标使用当前日期：YYYY-MM-DD-<name>
- 如果归档目标存在，该变更失败但继续处理其他变更
