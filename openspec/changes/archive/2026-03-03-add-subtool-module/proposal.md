## Why

当前工具箱仅支持本地操作，缺少调用外部独立进程工具的能力。通过添加抽象工具模块，工具箱可以在启动时自动扫描 `tools` 目录发现外部工具，将选中的设备信息传递给工具执行特定的运维操作并返回结果，实现更灵活的扩展。

## What Changes

- 新增抽象工具模块 `tool`，定义外部工具的统一抽象
- 工具在 `tools` 目录下按一级子目录组织，目录名即为工具 ID
- 每个工具目录下必须包含 `tool.json` 配置文件（定义 ID、名称、版本、启动命令等）
- 根据不同平台（Windows/Linux）使用不同的启动命令
- 工具箱启动工具时创建工作目录 `--work-dir` 并传递给工具
- 输入 JSON 包含完整设备信息（包括登录密码，当前明文，未来支持密文）
- 工具执行结果通过 result.json 返回，保存到工作目录
- result.json 包含 status（completed）和 result（success/failed）两个状态字段
- 运维结果保存到 `data/archive/{toolid}-{timestamp}` 目录（时间格式：20260303233450123）
- JSON 结果文件中时间字段格式：`2026-03-03 23:34:01.123`
- 运维过程中可能生成交付件（如报告等），由工具直接保存到该目录，交付件信息记录在 result.json 中
- 目录重复时追加递增序号
- 工具运行期间锁定所有选中的设备（由设备管理模块实现）
- 同一时间只能运行一个工具（全局锁）
- 工具箱通过轮询 result.json 中的 status 字段检测执行完毕
- **不提供**单独的手动注册工具接口（自动发现机制）
- 工具运行不考虑超时（可能运行相当长时间）
- 工具扫描、启动、结束操作均记录操作日志
- 添加模块内 UT 测试用例和模块级别集成测试用例

## Capabilities

### New Capabilities
- `tool`: 抽象工具模块，定义外部工具的自动发现和调用规范

### Modified Capabilities
（无）

## Impact

- 新增 `tool` 核心模块
- 影响范围：工具箱启动时加载层，需要与设备管理模块集成
- 需要设备管理模块提供设备锁定接口（lockDevices/unlockDevices）
